Brightspace Page Structure and Headings

Build pages that are easy to scan, navigate, and read with assistive technology. Use AI to draft clean structure, then validate before publishing.

AI can help draft headings, chunk content, and rewrite dense text. You are responsible for correct heading levels, accurate meaning, and validation in Brightspace.

Why this matters: clear structure helps students quickly find instructions and understand content using screen readers, keyboard navigation, mobile devices, and different reading strategies.

Fix or rebuild: choosing the most effective path

Brightspace content is most usable when it is authored as semantic HTML in the Brightspace editor. If a page is messy, rebuilding is often faster than patching repeated formatting problems.

  • Fix the page when the content is accurate and you mainly need to add headings, lists, clearer link text, and short spacing improvements.
  • Rebuild the page when it is long, unchunked, uses bold as headings, has pasted formatting from Word or PDFs, or relies on screenshots of text.

Both actions demonstrate a good-faith effort toward compliance and improve learning for everyone.

Reminder: structure is an accessibility feature

Brightspace pages: headings and structure

Typical uses include weekly modules, instructions, announcements, and learning activities. Good structure reduces cognitive load and improves navigation for screen reader and keyboard users.

How to draft or revise with AI (Brightspace HTML)
  1. Copy your current Brightspace page text into an approved AI tool. Do not include student identifiers.
  2. If the content includes tables, images, or links, include the relevant text around them so the AI can preserve meaning.
  3. Use one of the section prompts below to generate a revised version.
  4. Paste the revised content back into Brightspace and use the editor formatting tools (headings, lists, links).
  5. Run the Brightspace accessibility checker (if enabled) and then complete the manual checklist and validation steps.

Tip: after pasting, re-apply headings and lists using the editor controls

HTML beats pasted formatting

Pages that look fine after pasting from Word or PDFs can still be poorly structured under the surface. Rebuild or reformat in the Brightspace editor so headings and lists are real structure, not visual styling.

Common risk: pasted content that is bolded and spaced like headings but is not a heading

Headings and chunking

Headings are not visual decoration. They are navigation landmarks. Students using screen readers often jump by headings, and all students benefit from predictable chunks.

  • Use one clear page title (Heading 1) and then use Heading 2 for major sections.
  • Do not skip heading levels. Use Heading 3 only within a Heading 2 section.
  • Keep sections short. Break long pages into smaller pages when the topic shifts.
  • Do not use bold text as a substitute for headings.

Common error: multiple H1 headings on one page

What Ally will not catch

Ally can flag missing structure, missing alt text, and some formatting issues. It will not confirm whether your headings make sense instructionally, whether your page is appropriately chunked, or whether students can locate key instructions quickly.

Use Ally as a signal, then apply the checklist and a quick navigation check.

Before and after: what better structure looks like

Before: a long page with a single heading, dense paragraphs, and directions embedded in text. Students must scroll and search to find due dates, steps, and submission details.

After: a short page title followed by clear sections such as Overview, What to do, What to submit, and How you will be assessed. Steps appear as a numbered list, and key items are easy to scan on mobile.

Common misconception

If the text looks like a heading because it is bold and larger, it must be a heading. Visual styling alone does not create structure. Use the Brightspace heading format so assistive technology can navigate it.

Manual remediation checklist for Brightspace page structure
  • Use one page title (Heading 1). Use Heading 2 for sections. Use Heading 3 only within a section.
  • Make lists real lists (bullets or numbers), not typed hyphens or manual numbering.
  • Use descriptive link text that tells students where the link goes.
  • Keep paragraphs short and add spacing so the page is scannable.
  • Avoid images of text. Convert to real text or rebuild the content.
  • If you reference color or position ("in red," "on the right"), add a second cue (label, icon with text, or explicit wording).
  • Quick mobile check: confirm headings and key instructions are easy to find on a small screen.
Validation for Brightspace page structure outputs
  • Run the Brightspace accessibility checker for HTML-authored content, if it is available in your editor.
  • Run Ally Course Report for the page and resolve remaining structure issues.
  • Keyboard check: Tab through the page and confirm you can reach all links in a logical order.

Typical time: 5 to 20 minutes

Common pitfalls in Brightspace page structure
  • Using bold as headings, which does not create real structure.
  • Skipping heading levels, which breaks navigation.
  • Long pages with no chunking, causing students to miss key instructions.
  • Copy-paste formatting from Word or PDFs that looks fine visually but creates inconsistent structure.

Readability conventions that reduce barriers

Readable pages are more accessible. These conventions reduce confusion for students with disabilities and improve clarity for everyone.

Short paragraphs, clear lists, and consistent spacing
  • Prefer short paragraphs (2 to 5 lines).
  • Use lists for steps, requirements, and criteria.
  • Use consistent labels for repeated items (Due date, Time estimate, What to submit).
  • Use meaningful headings that match what students are looking for.
  • Write link text that is understandable out of context.

Tip: structure helps students scan on mobile and reduces missed instructions

Rewrite for clarity without changing meaning 
You are helping me revise course instructions for clarity and accessibility.

Revise the text to:
- Use short paragraphs and plain language
- Convert steps and requirements into real lists
- Add clear labels (for example: Due date, What to submit, How to submit)
- Keep the original meaning and course policies
- Do not add new requirements

Return plain text only.
Common pitfalls in readability
  • Walls of text with no headings or lists.
  • Directions embedded in paragraphs instead of steps.
  • Inconsistent terminology for the same task across weeks.
  • Link text like "click here" or "read more" with no context.

Keyboard access and visible focus

Accessibility is not only screen readers. Keyboard-only navigation is a separate requirement and a common student need.

Quick check: can you tab through it
  • Use Tab to move forward through interactive items (links, buttons, controls).
  • Use Shift + Tab to move backward.
  • Confirm you can always see where you are on the page.
  • Confirm any embedded tools are also usable by keyboard.

If an embedded item traps your keyboard, provide an alternative or rebuild.

Validation for keyboard and focus
  • Tab through the page and confirm all links are reachable in a logical order.
  • Confirm links and focus states are perceivable without relying on color alone.

Typical time: 2 to 8 minutes

Accessible template packages*

Some Brightspace environments support adding an accessible template package to course files. If your environment supports this, templates can speed up consistent page structure and reduce formatting errors.

  • Use templates to standardize headings, spacing, and content blocks across modules.
  • Do not use templates as a substitute for clear instructions and descriptive links.
