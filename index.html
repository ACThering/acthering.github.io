Accessible Media: Video, Audio, Summaries

AI speeds up captioning and transcript work but does not finish accessibility. Provide captions or a transcript, apply the prompt, then complete the manual checklist and validation steps.

Fix or rebuild: choosing the most effective path

Accessibility remediation for media often starts with a decision. In many cases, rebuilding or replacing a recording is more efficient and produces better outcomes than repairing poor source material. The goal is not to preserve the original recording at all costs but to provide content that is perceivable, understandable, and usable for all learners.

  • Fix the media when the audio is clear and captions or a transcript are mostly accurate with correctable errors.
  • Rebuild or replace when the audio is difficult to understand, captions are severely inaccurate, or the recording is outdated and no longer instructionally necessary.

Both actions demonstrate a good-faith effort toward compliance and support an inclusive learning environment.

Reminder: replacement is an accessibility strategy

Video with captions

Typical uses include lecture recordings, demonstrations, guest talks, and narrated slide videos.

How to upload and use the prompts (Video)
  1. Use an institutionally approved AI tool. Do not upload sensitive student information.
  2. Prepare your source text. Captions or a transcript work best. If needed, export captions from your video platform or copy the transcript text.
  3. Provide the content in one of these ways:
    • Upload a caption or transcript file if file upload is available.
    • Paste caption or transcript text into the AI tool if file upload is not available.
  4. Paste one prompt from the prompt bank below.
  5. Retrieve the result and review it while playing the video to confirm timing cues, key terms, and speaker labels.
  6. Complete the manual remediation checklist and validation steps.

Manual remediation checklist for Video
  • Ensure captions are present for required video.
  • Confirm caption accuracy for technical terms, names, and key concepts.
  • Confirm speaker identification when instructionally relevant.
  • Ensure non-speech audio that affects meaning is represented in text when appropriate.
  • Provide a transcript when captions are not available or not sufficient.
  • Ensure critical instructions are also available in text outside the video.
Validation for Video outputs
  • Play the video while reviewing captions to confirm alignment and accuracy.
  • If you post a transcript or summary in Brightspace, run the Ally Course Report for the page or document.
  • Spot check the posted Brightspace content with a screen reader.

Typical time: 10 to 30 minutes

Common pitfalls in Video
  • Auto captions that misrepresent technical terms and names.
  • Missing speaker labels in multi-speaker instruction.
  • Important information delivered only through visuals or audio with no text alternative.
  • Outdated recordings that are faster to replace than to repair.

Manual remediation checklist for Third party
  • Verify that captions or a transcript exist on the source platform. If not, consider replacing the item or posting a brief instructor summary near the link.
  • Label the link with the title, creator, and format, for example “Video” or “Podcast.”
  • Avoid autoplay and embedded players that add ads or trackers. Provide a standard hyperlink when needed.

Common pitfalls in Third party
  • Dead or geo-restricted links with no alternative.
  • Auto captions that are inaccurate for technical terms.

  • Summaries that introduce new information not present in the source.

  3. Provide the transcript by uploading a file or pasting the text, and include timestamps if available.

Prompt bank (Audio description)

Prompt A. Create an audio description script from a transcript

You may be receiving a transcript, captions, or pasted text from a video.

If readable transcript or caption text is present, write an audio description script that narrates key visual information not spoken in the audio. Use neutral, concise language in present tense. Include:
• Actions, scene changes, on‑screen text, charts, and gestures that affect meaning
• Speaker identification when not clear from audio
• Placement notes like [insert during pause] only when necessary
If timestamps are provided, align description to major time points. Do not invent timestamps.

If there is no readable transcript or captions, briefly state what content is needed to proceed.

Return the audio description script as plain text.

Prompt B. Student note for Brightspace and choice of description method

You may be receiving a URL plus transcript or captions.

Draft a short student note for Brightspace that explains how audio description is provided for this video. Include:
• Method used (built‑in narration, separate description track, or text description on the page)
• Where to find the description (for example, the "AD" audio track in the player or the text below the video)
• One sentence on instructional purpose
Do not add new facts beyond the source.

Return plain text only.
Manual remediation checklist for Audio description
  • Confirm that meaningful visual information is described when not available in audio.
  • Prefer built‑in narration during production when possible. Use a separate track or text description when rebuilding is not practical.
  • Ensure technical terms, labels on charts, and on‑screen text are included when they affect learning.
  • Keep language concise and objective. Avoid interpreting or teaching new content in the description.
  • Clearly label the presence and location of the description in Brightspace or the media player.
Validation for Audio description outputs
  • Play the video without looking at the screen and confirm that all critical visuals are understood from audio plus description.
  • If using a separate description track, test track selection in the media player.
  • If using a text description on the page, run the Ally Course Report and spot check with a screen reader.

Typical time: 10 to 30 minutes

Common pitfalls in Audio description
  • Descriptions that repeat dialogue instead of covering visual information.
  • Missing description for on‑screen text, charts, or title slides.
  • Descriptions that are too long to fit in natural pauses.
  • Unlabeled or hard‑to‑find description tracks in the player.

Brightspace Page Structure and Headings

AI can help draft headings, chunk content, and rewrite dense text. You are responsible for correct heading levels, accurate meaning, and validation in Brightspace.

Why this matters: clear structure helps students quickly find instructions and understand content using screen readers, keyboard navigation, mobile devices, and different reading strategies.

Fix or rebuild: choosing the most effective path

Brightspace content is most usable when it is authored as semantic HTML in the Brightspace editor. If a page is messy, rebuilding is often faster than patching repeated formatting problems.

  • Fix the page when the content is accurate and you mainly need to add headings, lists, clearer link text, and short spacing improvements.
  • Rebuild the page when it is long, unchunked, uses bold as headings, has pasted formatting from Word or PDFs, or relies on screenshots of text.

Both actions demonstrate a good-faith effort toward compliance and improve learning for everyone.

Reminder: structure is an accessibility feature

Brightspace pages: headings and structure

Typical uses include weekly modules, instructions, announcements, and learning activities. Good structure reduces cognitive load and improves navigation for screen reader and keyboard users.

How to draft or revise with AI (Brightspace HTML)
  1. Copy your current Brightspace page text into an approved AI tool. Do not include student identifiers.
  2. If the content includes tables, images, or links, include the relevant text around them so the AI can preserve meaning.
  3. Use one of the section prompts below to generate a revised version.
  4. Paste the revised content back into Brightspace and use the editor formatting tools (headings, lists, links).
  5. Run the Brightspace accessibility checker (if enabled) and then complete the manual checklist and validation steps.

Tip: after pasting, re-apply headings and lists using the editor controls

HTML beats pasted formatting

Pages that look fine after pasting from Word or PDFs can still be poorly structured under the surface. Rebuild or reformat in the Brightspace editor so headings and lists are real structure, not visual styling.

Common risk: pasted content that is bolded and spaced like headings but is not a heading

Headings and chunking

Headings are not visual decoration. They are navigation landmarks. Students using screen readers often jump by headings, and all students benefit from predictable chunks.

  • Use one clear page title (Heading 1) and then use Heading 2 for major sections.
  • Do not skip heading levels. Use Heading 3 only within a Heading 2 section.
  • Keep sections short. Break long pages into smaller pages when the topic shifts.
  • Do not use bold text as a substitute for headings.

Common error: multiple H1 headings on one page

What Ally will not catch

Ally can flag missing structure, missing alt text, and some formatting issues. It will not confirm whether your headings make sense instructionally, whether your page is appropriately chunked, or whether students can locate key instructions quickly.

Use Ally as a signal, then apply the checklist and a quick navigation check.

Before and after: what better structure looks like

Before: a long page with a single heading, dense paragraphs, and directions embedded in text. Students must scroll and search to find due dates, steps, and submission details.

After: a short page title followed by clear sections such as Overview, What to do, What to submit, and How you will be assessed. Steps appear as a numbered list, and key items are easy to scan on mobile.

Common misconception

If the text looks like a heading because it is bold and larger, it must be a heading. Visual styling alone does not create structure. Use the Brightspace heading format so assistive technology can navigate it.

Rewrite my page with correct headings and lists

Task:

2) Use one clear page title, then section headings that follow a logical hierarchy.
3) Convert any fake lists into real bullet or numbered lists.
4) Break long paragraphs into shorter paragraphs.
5) Preserve meaning and course specific terminology. Do not add new facts.

Output requirements:
- Provide the revised content in plain text with heading labels such as:

  [H2] Section
  [H3] Subsection
- Keep it ready to paste into Brightspace.
Fix bold-as-heading and heading level issues 
You are helping me fix common heading problems in a Brightspace page.

Identify:
- Any lines that are visually styled like headings (for example bold text) but should be true headings
- Any skipped heading levels (for example going from H1 to H3)
- Any repeated H1 headings that should be H2 instead

Then:
- Propose a corrected heading outline (H1, H2, H3)
- Provide a revised version of the page content using that corrected outline
- Do not change meaning and do not add new facts

Return plain text only with [H1], [H2], [H3] labels.
Manual remediation checklist for Brightspace page structure
  • Use one page title (Heading 1). Use Heading 2 for sections. Use Heading 3 only within a section.
  • Make lists real lists (bullets or numbers), not typed hyphens or manual numbering.
  • Use descriptive link text that tells students where the link goes.
  • Keep paragraphs short and add spacing so the page is scannable.
  • Avoid images of text. Convert to real text or rebuild the content.
  • If you reference color or position ("in red," "on the right"), add a second cue (label, icon with text, or explicit wording).
  • Quick mobile check: confirm headings and key instructions are easy to find on a small screen.
Validation for Brightspace page structure outputs
  • Run the Brightspace accessibility checker for HTML-authored content, if it is available in your editor.
  • Run Ally Course Report for the page and resolve remaining structure issues.
  • Keyboard check: Tab through the page and confirm you can reach all links in a logical order.

Typical time: 5 to 20 minutes

Common pitfalls in Brightspace page structure
  • Using bold as headings, which does not create real structure.
  • Skipping heading levels, which breaks navigation.
  • Long pages with no chunking, causing students to miss key instructions.
  • Copy-paste formatting from Word or PDFs that looks fine visually but creates inconsistent structure.

Readability conventions that reduce barriers

Readable pages are more accessible. These conventions reduce confusion for students with disabilities and improve clarity for everyone.

Short paragraphs, clear lists, and consistent spacing
  • Prefer short paragraphs (2 to 5 lines).
  • Use lists for steps, requirements, and criteria.
  • Use consistent labels for repeated items (Due date, Time estimate, What to submit).
  • Use meaningful headings that match what students are looking for.
  • Write link text that is understandable out of context.

Tip: structure helps students scan on mobile and reduces missed instructions

Rewrite for clarity without changing meaning 
You are helping me revise course instructions for clarity and accessibility.

Revise the text to:
- Use short paragraphs and plain language
- Convert steps and requirements into real lists
- Add clear labels (for example: Due date, What to submit, How to submit)
- Keep the original meaning and course policies
- Do not add new requirements

Return plain text only.
Common pitfalls in readability
  • Walls of text with no headings or lists.
  • Directions embedded in paragraphs instead of steps.
  • Inconsistent terminology for the same task across weeks.
  • Link text like "click here" or "read more" with no context.

Keyboard access and visible focus

Accessibility is not only screen readers. Keyboard-only navigation is a separate requirement and a common student need.

Quick check: can you tab through it
  • Use Tab to move forward through interactive items (links, buttons, controls).
  • Use Shift + Tab to move backward.
  • Confirm you can always see where you are on the page.
  • Confirm any embedded tools are also usable by keyboard.

If an embedded item traps your keyboard, provide an alternative or rebuild.

Validation for keyboard and focus
  • Tab through the page and confirm all links are reachable in a logical order.
  • Confirm links and focus states are perceivable without relying on color alone.

Typical time: 2 to 8 minutes

Color and Contrast: Readability and Meaning

Color and contrast affect whether learners can perceive, read, and interpret content. Adequate contrast helps in low-vision and color-blind contexts and improves readability on mobile devices, projectors, and in bright environments. Maintain strong contrast to reduce eye strain and keep critical information visible to everyone.

WCAG references: 1.4.3 Contrast (Minimum), 1.4.11 Non-text Contrast, 1.4.1 Use of Color

Choose the most effective change

Adjusting color values is not the only path. Often the smallest change reaches the target ratio while preserving brand feel. If color conveys meaning, add a second cue.

  • Change color values to meet WCAG 2.1 AA.
  • Increase size or weight so large text can pass at 3:1.
  • Add an overlay when text sits on an image.
  • Add a non-color cue when color signals meaning.

Reminder: do not rely on color alone for meaning

Suggest accessible color fixes

Typical uses include Brightspace HTML pages, PowerPoint slides and banners, tables, and PDFs that use UB brand colors or custom color schemes.

How to apply these checks (Color Fixes)
  1. Gather exact hex values for foreground and background for text, links, buttons, banners, and form controls. Sample the actual rendered color including overlays and transparency.
  2. Determine whether to adjust text, background, or font size and weight. Large text qualifies for a lower ratio.
  3. Run a contrast check and record the measured ratio. Text must be at least 4.5:1 for normal text. Large text can pass at 3:1.
  4. Large text definition: 18 pt (24 px) or larger regular, or 14 pt (18.66 px) and bold.
  5. Choose the smallest visual change that reaches WCAG 2.1 AA and preserves brand intent. Recheck after theme or background changes.
AI-assisted checks and adjustments (Color Fixes) 
You may be receiving hex values for text, links, buttons, and backgrounds.
Suggest the smallest adjustments that reach WCAG AA while preserving brand intent.
Return revised hex values, the measured ratios, and one sentence of rationale per element.

Use a reliable checker such as the WebAIM Contrast Checker to confirm final ratios.

Manual remediation checklist for Color Fixes
  • Normal text meets 4.5:1. Large text meets 3:1.
  • UI components and focus indicators meet 3:1.
  • Text over images uses a solid or semi-opaque overlay.
  • Final colors preserve legibility and brand feel.
Validation for Color Fixes outputs
  • Confirm ratios with a contrast checker and record values. If tools disagree, record the measured hex values and use a trusted checker for final decisions.
  • Run the Ally Course Report in Brightspace for posted pages.
  • Spot check with a screen reader and keyboard navigation to confirm visible focus.

Typical time: 5 to 20 minutes

Common pitfalls in Color Fixes
  • Light gray text on light backgrounds.
  • Headings on photo banners without an overlay.
  • Changing default state colors but missing hover or focus states.
  • Assuming tools always agree without context.

Identify color-only meaning and add a second cue

Typical uses include charts, graphs, buttons, alerts, required-field indicators, quizzes, and feedback messages that use color to indicate meaning.

How to apply these checks (Visual Cues)
  1. Scan for places where color alone signals meaning or state.
  2. Note affected elements such as charts, legends, status messages, and required fields.
  3. Add one extra cue per case such as an icon, label, underline, pattern, or direct data labels.
  4. Recheck clarity on mobile and when printed in grayscale.
AI-assisted checks and adjustments (Visual Cues) 
You may be receiving a description of components where color signals meaning.
Identify each instance of color-only meaning and propose a second cue:
• Icons or short text for status
• Patterns or direct labels for charts
• Underlines or bold for links
Return concise recommendations that fit the current layout.

For design ideas that work well for color vision differences, see guidance from Perkins School for the Blind: How to design for color blindness.

Manual remediation checklist for Visual Cues
  • Links are not differentiated by color alone. Prefer underlines or a strong non-color cue.
  • Status messages include icons or short text.
  • Charts use patterns or direct labels, not color alone.
  • Required fields include the word “Required.”
Validation for Visual Cues outputs
  • Preview with a color-blindness simulator or design guide and also test in grayscale or print to PDF.
  • Confirm icons and labels remain visible at small sizes.
  • Run the Ally Course Report for posted pages and spot check with a screen reader.

Typical time: 10 to 25 minutes

Common pitfalls in Visual Cues
  • Red and green used without labels.
  • “Important items in blue” with no supporting cue.
  • Heatmaps or scales with indistinguishable steps for some viewers.

Link color vs surrounding text

Typical uses include Brightspace pages, Word and PDF documents, and email content with hyperlinks and visited states.

How to apply these checks (Link Contrast)
  1. Evaluate links in context such as body text, sidebars, banners, and dark themes.
  2. Check link contrast against page background and distinction from body text.
  3. Confirm hover, focus, and visited states remain accessible.
AI-assisted checks and adjustments (Link Contrast) 
You may be receiving hex values for body text, link text, and backgrounds.
Recommend a link color that meets contrast requirements and is clearly distinct from body text.
Include guidance for underline, hover, focus, and visited states. Return rationale and measured ratios.

Confirm final ratios using the WebAIM Contrast Checker. Ensure visited link colors also pass.

Manual remediation checklist for Link Contrast
  • Link color has sufficient contrast with background.
  • Links are visually distinct from body text, preferably underlined.

Images: Alt Text, Long Descriptions, and Decorative Images

AI can draft alt text and long descriptions, but it does not determine what is instructionally essential. Provide purpose and context, review carefully, and validate in Brightspace.

Fix or rebuild: choosing the most effective path

For images, accessibility remediation often starts with a decision. In many cases, replacing an image with accessible text, a simple table, or a redesigned visual is more effective than attempting to describe a complex or low-quality image. The goal is not to preserve the original image at all costs but to ensure learners can access the instructional meaning.

  • Fix the image use when it is instructionally meaningful, visually clear, and can be described accurately with alt text or a long description.
  • Rebuild or replace when the image contains dense text, low-resolution content, complex charts without underlying data, or decorative styling that obscures meaning.

Reminder: replacement is an accessibility strategy

Decorative images: choosing the right approach

Not every image needs alt text. Images that do not convey instructional meaning should be marked as decorative so they are skipped by screen readers.

Mark as decorative when the image does not convey instructional meaning. The image can be removed without affecting what students are expected to learn.

Write alt text when the image communicates essential information that can be conveyed accurately in one or two sentences.

Provide a long description when the image is complex and cannot be adequately described in one or two sentences, such as charts, diagrams, or multi-part visuals.

Decorative, alt text, or long description: quick decision guide
If the image Then choose What to do
Does not add instructional meaning and is not referenced in instructions Mark as decorative Check “decorative” so screen readers skip the image. In most learning platforms, this is preferable to leaving alt text blank.
Conveys instructional meaning that can be explained briefly Alt text Write 1 to 2 concise sentences describing the essential information
Is complex or multi-part and cannot be explained briefly Long description Provide alt text plus a longer text description placed near the image

Choosing the simplest option that fully communicates the instructional meaning is the most accessible approach.

Common examples
  • Decorative: Stock photo banner used for visual interest only
  • Decorative: Icon that repeats adjacent text (for example, a calendar icon next to a due date)
  • Alt text: Screenshot that shows where to click to submit an assignment
  • Long description: Multi-part diagram or infographic where relationships and labels matter
Common pitfalls
  • Adding alt text to decorative images, which creates unnecessary repetition and distraction for screen reader users
  • Marking an image as decorative when students are expected to interpret it
  • Using long descriptions for simple images that can be described in one sentence

Tables and Data: Accessible Structure and Use

AI can assist with drafting table structures and summaries, but authors remain responsible for clarity, accuracy, and instructional intent. Always validate headings, table headers, captions, and reading order in the final delivery format.

Decision guide: "Should this be a table?"

Tables support accessibility when they help learners compare values, identify relationships, or scan structured data. If you do not need comparisons or relationships, write the content as text instead. Rebuilding a table as clear text is often the most accessible option.

Choose the simplest format that supports learning and accessibility
If the content Use What to do
Requires comparing values across rows or columns Data table Use a real table with a clear header row (and header column if needed), plus a caption.
Lists items without comparison (for example, steps, supplies, policies) Text with headings and lists Write it as text using headings and bulleted or numbered lists.
Is only arranged in columns for layout or visual alignment Text layout (not a table) Rebuild as headings, paragraphs, and lists. Do not use tables for layout.
Is highly complex with merged cells or nested structures Rebuild Simplify into multiple smaller tables or convert key information into text and lists.

Quick check

Read each row out loud as if a screen reader is reading it: header, then each cell. If it does not make sense in that order, rebuild the table or switch to text.

Prompt B. Convert a table to accessible Brightspace HTML (preferred for complex tables)

You may receive an uploaded Word document or pasted table content.

Convert the table into semantic HTML for Brightspace:
- Include a <caption>
- Use <thead> and <th scope="col"> for column headers
- Use <th scope="row"> if a row header is needed
- Avoid merged cells unless essential

Return HTML code only, then a short validation checklist.
Manual checklist (Word tables)
  • Use a real table only when learners must compare values or interpret relationships.
  • Keep tables simple: one header row, no merged cells when possible.
  • Write a short caption above the table (Word does not provide an HTML-style caption element).
  • Confirm any abbreviations are defined near the table.
  • Run Review, then Check Accessibility.
  • If the table is long, set the header row to repeat so screen readers and learners retain context.

Microsoft support: Repeat table header on subsequent pages

Simplified guidance

Check the option to repeat the header row so the header remains associated with the table content. This supports accessibility and reduces confusion when tables continue.

Validation (Word tables)
  • Run Microsoft Accessibility Checker in Word.
  • Confirm the table reads logically from headers to each row.
  • Consider converting complex tables to Brightspace HTML for better usability.

Typical time: 10 to 30 minutes

Data from Excel: publish for reading, not spreadsheets

Excel is powerful for analysis, but many spreadsheets are difficult to deliver accessibly as course content. For most courses, the highest payoff is to publish the data learners need in an accessible table (HTML or Word) plus a short summary, instead of posting a complex spreadsheet.

How to upload and use the prompts (Excel data)
  1. Use an institutionally approved AI tool when working with university or student-related content.
  2. Remove student identifiers. If the workbook includes grades or student work, do not upload it.
  3. Upload the .xlsx file or paste the relevant table range with headers.
  4. Paste one prompt below.
  5. Review for accuracy, especially labels, units, and totals.
  6. Publish the result in Brightspace and validate with Ally.

Writing Clear Descriptive Links to Webpages and Resources

Descriptive link text tells learners where a link goes and what to expect before they activate it. Screen reader users often navigate long pages by reviewing links out of context, so vague or generic link text limits comprehension and slows learning.

AI and author responsibility

AI can assist with drafting link text, but authors remain responsible for clarity, accuracy, and instructional intent.

Choosing descriptive link text decision guide

This applies when linking phrases like “click here,” “this page,” or entire sentences instead of naming the destination clearly.

Descriptive link text: quick decision guide
If the link is Then choose What to do Example
A URL pasted into the page Descriptive title Use the destination title or a meaningful label instead of the raw URL. Accessibility at UB: Creating Accessible Word Documents
Generic text such as “click here” or “this page” Purpose plus destination Name the destination and what the learner will do or find there. Submit your reflection in Brightspace Assignments
The title of a webpage or document Exact or shortened title Match the destination title or a shortened version that preserves meaning. Avoid truncating key words. Section 508: Descriptive Links and Hypertext
The same resource appears multiple times Consistent link text When the same resource appears in multiple places, use the same link text each time. Course Accessibility Checklist
A task with a clear action Action plus object Make the action explicit so learners know what happens when they activate the link. Download the lab data template (Excel)

Most learners benefit from clear link text, but this practice is especially critical for students using screen readers or keyboard navigation.

Quality checks you can do in under one minute
  • Out-of-context check: Read each link by itself and confirm it still makes sense.
  • Scan check: Skim only the linked phrases on the page and confirm you can understand the workflow without reading surrounding text.
  • Consistency check: Confirm repeated resources use the same link text across modules and weeks.
Manual checklist for descriptive links
  • Use meaningful link text that describes destination and purpose.
  • Keep link text concise while preserving key words.
  • Avoid using only “click here,” “more,” “this,” or the raw URL as the link text.
  • If the link initiates a task, include the action in the link text.
  • If the link points to a file, include the file type when helpful (PDF, Word, PowerPoint, Excel).

Reminder

Validation in Brightspace
  • Run the Ally Course Report for the page and review any link related feedback.
  • Keyboard check: Use Tab to move through links and confirm focus is visible and the link order matches the page structure.
  • Screen reader spot check: Navigate by links and confirm the link list is understandable without surrounding context.

Typical time: 2 to 10 minutes per page

Common places faculty get stuck
  • Copying and pasting a full URL because it feels faster than naming the destination.
  • Linking an entire sentence instead of a concise destination phrase.
  • Using different names for the same resource across weeks, which confuses learners who skim.
  • Using link text that implies the wrong action (for example, “Download” when the link opens a webpage).
